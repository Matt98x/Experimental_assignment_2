4 ## Introduction
5 
6 This assignment target is to build an ROS architecture to implement a robot, simulating a pet, that
7 interact with a human and moves in a discrete 2D environment.
8 The pet has three states representing behaviours, that are simulated as a finite state machine with 3 states:
9 * Play
10 * Sleep
11 * Normal
12 
13 These states determine the way the robot act inside the grid, whether moving randomly as in normal, going to targets determined by the position of a ball guided by the human or simply sleeping in the home position.
14 
15 The robot will change between states randomly, eccept for play, which is received by the user indirectly: by imposing a positive height with respect to the ground to the ball which can now be seen by the robot and activate the play routine.
16 
17 With respect to the first assignment, with the new simulation environment, both the world and the pet have been modified:
18 * The pet has is composed by a 2 weeled robot, with balance handled by a sphere(spherical wheel), a long neck in the front and a rgb camera which can oscillate by 90Â° in both directions, resembling a stylized dog. The movement is handled by a differential drive.

22 * The pet world is a delimited planar surface of dimension 16x16 with center at 0x0, the only objects inside the map are the user(a person sitted on a chair), the ball(a green sphere) and the pet, there can be no obstacles between the robot and the ball, which means that obstacle avoidance algorithms are not required.

28 
29 ### Architecture
30 The software architecture consists of two main areas: the world and the pet, programmatically represented by the two packages that compose this implementation.
31 The first handle how the world is organized and works externally to the pet. This means that the commander, the Pet_logic(now simply interpretable as Logic) and the movement of the ball can ba all thought to be component of this macro_architecture.
32 Going to the Pet, this can be considered as connected to the world over just two aspect:
33 * The Perception: that handles how the pet perceives the world and in particular the ball
34 * The Logic: which is simply a remaining part from the first assignment, in which this node controlled both the command interpretation and the control of the robot state.

40 Going in depth of the components, we have:
41 * Random command generator(Command_giver.py): randomically send a string representing the concatenation of one or more commands of the form: 'play'(to start the play state),'point to x y'(to simulate the pointing gesture to x y),'go to x y'(to simulate the voice command to x y) and 'hide'(to stop the play state without entering the sleep state)
42 * Interpreter(Pet_logic.py): to interpret the string commands and translate them to movement of the ball, moreover, it handles the switch from play and normal to sleep and from sleep to normal
43 * Pet behaviours(Pet_behaviours.py): that simulate behaviours as a finite state, in the already mentioned states
44 * Perception(robot_following.py): which is the node that handles the camera inputs(target identification and research) and the hardware control for these tasks(control of the neck joint(target tracking) and body(target search) )
45 * Actor: may or may not be present and provides the same type of messages that the Command_giver provides, adding also symbolical location such as "home" and "owner", moreover can query or set the state of the robot and interact with the parameters.
46 * Ball: representation of the logic controlling the ball
47 * Robot Control: representation of the low_level control of the ball
48 * Gazebo: although not a logic part of the component diagram, it represent the way ball, robot control and perception are fundamentally part of the simulation and communicate with it

55 ### State Machine
56 Now, we can discuss the finite state machine. This, can be described by the following image:
57 
58 \image html general_state_machine
59 <center>Finite state machine diagram</center>
60 
61 The Normal state is the simplest in nature of the three states, it simply consist of a loop of setting random destinations inside the grid without other interventions while the targets are not achieved.
62 
63 On the other hand, the sleep consist in setting the target to 'home' (set in the parameter server), and, when the position is achieved, just wait ignoring all signals exept for the change of state.
64 
65 While the 'Sleep' and 'Normal' state are quite simple in nature, the 'Play' state is quite more complex in nature.
66 
67 Of course, having to consider the position of the ball without having it, we have to construct a control with the few notions we have: the relative radius of the ball and the angle of the neck with respect to the principal axis of the chassis(that is the principal axis of the robot).
68 
69 With this, the algorithm is to first minimize the angular offset of the neck w.r.t. the body rotating the body itself and then set a linear velocity while you receive the two data from the Perception node.
70 
71 Obviously, while this process is happening, the state is checked and change if the change conditions are satisfied.
72 
73 Although this is the part of the state explicitely coded in the finite state machine, an additional part is present inside the Perception node and is related to how the target is handled and searched.
74 
75 When the target is achieved, the robot proceeds to what has been defined as "swing routine", in which it moves the neck 45Â° to the left and to the right before centering back again to the target.
76 
77 When, eventually, the robot loose sight of the ball (always inside the perception code), the robot align the camera to the body and start spinning in the same direction of the last velocity of the body(right if the velocity was to be zero), to try and find it back again, if it cannot manage it, it will switch to the normal state.

81 The main parameters used for this implementation are:
82 * ball/ball_description: the gazebo description of the ball
83 * /home x&y: coordinates of the home, location where the pet sleeps
84 * /human_description: gazebo description of the human model
85 * /robot/camera1: set of parameters of the camera sensor
86 * /robot/joint1_position_controller: parameters for the neck joint
87 * /robot/joint_state_controller: parameters for the robot
88 * /robot/robot_description: gazebo robot description
89 * /state: state of the pet(play, normal, sleep)
90 * /gazebo: gazebo anvironment parameters
91 
92 Regarding the messages, they will be listed as
93 * std_msgs.String: used by the commander to the logic, in order to send the command for the ball
94 * std_msgs.Float64: used by the Perception and Behavious to control the neck joint
95 * geometry_msgs:Pose2D: used by Perception to send the radius and camera angle for the control in the play behaviour
96 * geometry_msgs:Twist: used by Behaviours and Following to Gazebo in order to control the twist of the robot
97 * sensor_msgs:JointStates: Used to read the values of the robot states
98 * sensor_msgs:Image: Message with the image camera information
99 * exp_assignment.PlanningAction: message of the action server, it is used by the Pet_logic and Behaviour to use the action server of the ball and robot respectively

200 ## Working assumptions
201 
202 The working assumptions will be discussed as the following list:
203 * The robot, simulating a pet, interact with a human with a ball moving in the environment and moves in a 2D surface in a simulation environment.
204 * Both the robot targets and its positions belongs exclusively to the map(16 by 16 grid with center in (0,0))representing the 2D environment.
205 * The robot has 3 main states:
206  - Play
207  - Normal
208  - Sleep
209 * The logic receive forms in strings with possible form:
210  - "play"
211  - "hide"
212  - "go to x1 y1" (equivalent to voice command)
213  - "point to x1 y1" (equivalent to pointing commands)
214 * Once logic receives the message it applies it to the ball
215 * if the command is "play", the ball is positioned above ground, if "hide", it is positioned below the ground .
216 * The robot activates the play mode only when the robot perceives the ball.
217 * When the ball is percieved the robot tries to get to it
218 * When achieved the "swing routine" is initialized, for which the neck is first angled to pi/4 and then too -pi/4 and then back to center
219 * If the ball is not in sight anymore, the robot procede to a full rotation to find if the ball is still in the environment, if not, it switch to the normal state..
220 * Two predifined positions inside the map are "Owner" and "Home", which cannot be changed during the execution, and can be used instead of coordinates in giving commands.

222 ## System features and limitations
223 
224 Starting from the limitations:
225 * The system is not scalable in the number of individually controllable robots, but if all robots have the same state, it is scalable, even if collision between robots are not handled
226 * It is not scalable in the number of symbolic locations
227 * It is not really scalable in the number of states
228 * Does not distinguish between the pointing action and the vocal command, since the ball act
229 * The robot control is not too responsive, since the control can't have a too high gain to avoid instability and the robot toppling over.
230 * The movement speed is high but make a trade-off for instability
231 * There are some problems with the target reaching when it is on the side of the robot especially when really close to the chassis(It cannot handle small curvature radii and the robot tends to run in circles around the target)
232 * Underline jittering in the motors, observable when control is not yet active
233 
234 Going on to the features:
235 * The robot is controlled in almost its entirety by the pet package, which means a high scalability and modularity
236 * We have the robot perspective in a separate window
237 * Can show the location of the robot in the map
238 * The robot can check the state without being stuck in an action server loop

241 ## Possible technical improvements
242 
243 There are many possible technical improvements to this architecture:
244 * Modify the simulation component to make it more scalable, introducing the state change from and to sleep inside the pet_package
245 * Improve the control, of both the camera and the robot chassis in such a way to perform both linear and angular velocities, and in a way that the robot do not topple over
246 * Handle the situation when the ball is close to the side of the robot, in order to have it centered inside the robot perspective
247 * Add a way to avoid collisions with other obstacles, with the possible introduction of a proximity sensor of some sort
248 * Add multiple robots to the simulation
