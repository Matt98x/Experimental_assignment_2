5 ## Introduction
6 
7 This assignment target is to build an ROS architecture to implement a robot, simulating a pet, that
8 interact with a human and moves in a discrete 2D environment.
9 The pet has three states representing behaviours, that are simulated as a finite state machine with 3 states:
10 * Play
11 * Sleep
12 * Normal
13 
14 These states determine the way the robot act inside the grid, whether moving randomly as in normal, going to targets determined by the user or simply sleeping in the home position.
15 
16 The robot will change between states randomly, eccept for play, which is received by the user.

28 The main components are:
29 
30 * Random command generator(Command_giver.py)- send a string representing the concatenation of one or more commands of the form: 'play'(to start the play state),'point to x y'(to simulate the pointing gesture to x y),'go to x y'(to simulate the voice command to x y)
31 * Pet Interpreter(Pet_logic.py)- to interpret the string commands and translate them to a command list
32 * Pet behaviours(Pet_behaviours.py)- that simulate behaviours as a finite state, in the already mentioned states
33 * Turtle simulation- that represents the position of the robot in the map
34 * User- may or may not be present and provides the same type of messages that the Command_giver provides, adding also symbolical location such as "home" and "owner"
35 
36 Starting from the Command_giver, it is a publisher, with String type message, that transmit a series of 1 to 5 commands as the one discussed with a conjunction of an "and".
37 
38 Talking about the Pet interpreter, this component subscribes to the command generator topic and provides a service with share the value of the command string composed by integers and devided by the '|' character.

77 ## Working assumptions
78 
79 The working assumptions will be discussed as the following list:
80 * The robot, simulating a pet, interact with a human and moves in a discrete 2D environment.
81 * Both the robot targets and its positions belongs exclusively to the map(11 by 11 grid)representing the 2D environment.
82 * The robot has 3 main states:
83  - Play
84  - Normal
85  - Sleep
86 * The robot receive forms in strings with possible form:
87  - "play"
88  - "go to x1 y1" (equivalent to voice command)
89  - "point to x1 y1" (equivalent to pointing commands)
90  - combination of commands with conjuctions of "and":
91  - All the command after the play are executed
92  - if a "play" is not in first place, only commands after the "play" command are executed
93 * The robot can receive any command while executing one in Play state but the ones given are neither executed nor stored.
94 * The robot can receive any command while in sleep state but the ones given are neither executed nor stored.
95 * Sleep preempt any other state when it starts.
96 * From Sleep you can only transition to Normal.
97 * The only command that can be received in Normal is "play".
98 * Two predifined positions inside the map are "Owner" and "Home", which cannot be changed during the execution, and can be used instead of coordinates in giving commands.

100 ## System features and limitations
101 
102 Starting from the limitations:
103 * The system is not scalable in the number of type of commands
104 * It is not scalable in the number of symbolic locations
105 * It is not really scalable in the number of states
106 * Does not provide a complete graphical interface, as the grid is not visible
107 * The simulation is not modifiable as it was out-sourced
108 * Does not distinguish between the pointing action and the vocal command
109 
110 Going on to the feature:
111 * Understand both integer and symbolic location, provided they are of the predefined nature
112 * Can show the location of the robot in the map, provide, via shell, the state transition and the commands generated by the command generator
113 * Can take any number of commands, even if the execution cannot be stopped if not by the random intervention by Pet_logic.py

116 ## Possible technical improvements
117 
118 There are many possible technical improvements to this architecture:
119 * Modify the simulation component to make it more scalable
120 * Modify the interpreter to broaden the symbolic targets( to do it, the method is to setup a search in the parameters server to extract the coordinates related to the string)
121 * Create a more comprehensive propositional logic, adding an 'or' conjunction to make the system more intelligent
122 * Add other states to the state machine, which implies also a modification of the Pet_logic.py
123 * Make distinction between the pointing action and the vocal command
